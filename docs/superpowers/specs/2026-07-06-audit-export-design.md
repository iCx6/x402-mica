# Audit export (CSV/JSON) + date filter — design

Date: 2026-07-06 · Status: approved

## Goal

Make the audit log deliverable, not just viewable: download it as CSV or JSON from
the existing `GET /audit` dashboard, filtered by date range — the artifact an
accountant or a DAC8/CARF-minded compliance officer actually asks for. The HTML
view gets the same date filter so the screen shows exactly what the download
contains.

## API

All on the existing `GET /audit` route (same key gate: `x-api-key` header or `?key=`):

| Param | Values | Behavior |
|---|---|---|
| `format` | absent / `csv` / `json` | absent → HTML view (as today); otherwise a download. Unknown value → 400. |
| `from` | `YYYY-MM-DD` or full ISO timestamp | inclusive lower bound on `ts`. Optional. |
| `to` | `YYYY-MM-DD` or full ISO timestamp | inclusive upper bound; a date-only value covers that whole day. Optional. |
| `page` | integer | HTML view only; ignored by exports (exports return the full filtered set). |

Malformed `from`/`to` (not `YYYY-MM-DD` or ISO timestamp shape) → 400.

## Design

1. **Date filtering (shared by HTML + export).** Every `ts` is generated by us via
   `new Date().toISOString()`, so the column is uniform fixed-width ISO text and
   filtering is a lexicographic SQL comparison: `ts >= :from AND ts <= :to`.
   A date-only `to` is expanded to `<to>T23:59:59.999Z` (exact end-of-day for our
   ts format). A date-only `from` needs no expansion (`"2026-07-06" <= "2026-07-06T…"`).
   Implemented as a pure helper returning the SQL fragment bounds, unit-tested.
2. **CSV export.** Header `ts,network,asset,amount,payer,tx_ref,mica_compliant`,
   RFC 4180 quoting (quote fields containing `"` `,` or newline, double inner
   quotes), **plus formula-injection guard**: any field starting with `=`, `+`,
   `-`, `@`, tab or CR gets a leading `'`. The payer value originates from an
   attacker-influenceable request header and the file's target audience opens it
   in Excel — this is a trust boundary, not polish. Pure function
   `toCsv(rows): string`, unit-tested.
3. **JSON export.** Array of row objects with `mica_compliant` as a real boolean.
4. **Download headers.** `Content-Disposition: attachment; filename="x402-mica-audit[-<from>][-<to>].csv|json"`;
   `Content-Type: text/csv; charset=utf-8` / `application/json`.
5. **Exports are unpaginated** — the full filtered set in one response. SQLite-scale
   data; if it ever outgrows this, switch to streaming (`ponytail:` comment marks
   the ceiling).
6. **HTML view filter UI.** A plain server-rendered `<form method="get">` above the
   table: two native `<input type="date">` fields (no JavaScript), hidden `key`
   field, Apply button, and `CSV` / `JSON` download links that carry the active
   `from`/`to`/`key`. Pager links carry `from`/`to` too. Row count and pager
   reflect the filtered total.

## Error handling

- 503 (dashboard disabled) and 401 (bad key) behave exactly as today, for exports too.
- 400 with a one-line plain-text message for unknown `format` or malformed `from`/`to`.

## Testing

- Unit (no network, in `dashboard.test.ts`): `toCsv` quoting + formula guard
  (incl. a payer like `=HYPERLINK(...)`), date-bound expansion helper
  (date-only vs full ISO, end-of-day), filename builder if non-trivial.
- Manual smoke test on the running demo server: HTML with filter, CSV download,
  JSON download, 400 on bad input, 401 without key.

## Out of scope (deliberate)

- XLSX output, streaming/async export, column selection.
- Filtering by any field other than date (asset, payer, network).
- Auth changes — the existing single API key stays.
